## Element vs assessment widget (drop mechanism)

Wordsearch is an **element**, not an assessment widget. `wordsearch` is **not** in
`support/module-setup.ts` `ASSESSMENT_DATA_TYPES`, so `resolveDropTarget` returns
`'canvas'`. The component is therefore dropped onto the
**Instructions-for-Students** page (`#desktop_view`) via the existing
`cy.dropComponent('Word Search')` primitive, which matches the catalog widget
`{ name: "Word Search", dataType: "wordsearch" }` by **name** and injects it onto
the unrestricted canvas. It is **not** dropped into a Generic Step column.

`cy.ensureWordsearchDeepSetup` mirrors the **Text** element sibling's setup
contract: `cy.openAuthoringTool` → `cy.navigateToCanvasPage()` →
`cy.dropComponent('Word Search')`.

## Architecture notes (source-verified)

- **Canvas wrapper:** `.component-holder.wordsearch[wordsearch-template]`. There is
  **no `data-type` attribute** on the wrapper, so the **directive attribute**
  `[wordsearch-template]` is the stable selector hook.
- **Editable fields** (contenteditable): `#headerTextInput` (`fieldData.introduction`),
  `#instructionTextInput` (`fieldData.instruction`), `#questionTextInput`
  (`fieldData.question`) — set via `cy.setContentEditableOn`, never `cy.type()`.
- **Hidden Word entry** `#hiddenWordInput` is contenteditable (ng-model `textData`)
  with `pattern="[a-zA-Z]+"` + `ng-pattern-restrict`; `onKeyPress` **blocks spaces**
  (charCode 32), **stops typing past 13 chars**, and **triggers Add on Enter**.
  So **numbers, spaces, and >13-char words are rejected at input time** — these
  negative cases are verified against the input contract rather than by attempting
  (impossible) bad additions.
- **ADD** (`.add-btn` → `onAddButtonClick`) pushes `scope.textData.toUpperCase()`
  into `fieldData.settings.wordList`: words are stored **UPPERCASE**
  (case-insensitive), **de-duplicated** via `$.inArray`, and capped at
  `scope.maxWordCount = 13` words. `cy.addWordsearchWord(word)` drives this exact
  path through the directive scope (DOM-assertable result in `.wordListRepeat`).
- **WORDS section:** `.wordListHolder` → repeated `.wordListRepeat` rows
  (`.wordData` text + `.deleteWord` button → `onDeleteBtnClick`). The delete button
  is **`ng-disabled` when `wordList.length == 1`** — the WORDS section cannot be
  fully emptied from the author DOM. **Show Words** toggles
  `fadeInWords`/`fadeOutWords` (visibility only — no DOM removal).
- **Settings panel** `#word-settings-panel` (title "Word Search") exposes
  default-checked checkboxes `#add-show-me` / `#header-visibility` /
  `#instr-visibility` / `#showWrds-visibility`, a **Grid "Shuffle"** button
  (`.shuffleBtn`), outline radios `input[name="outline"]` (`outline` | `outlineBg`),
  and a colour picker `.colorPickerInputBox` (default `#7eb1eb`).
- **Grid** (`.puzzle-holder` → `.puzzleSquare`, `.addedWord` highlight), the **blue
  word-line** highlight, and the solve **strikethrough** are produced by
  `wordfind.js` at runtime — deferred.

## Why these deferrals

The wordsearch **grid** and all highlight/strikethrough feedback are generated by
`wordfind.js` at runtime and are most meaningfully asserted in **Preview/Reader**,
not the authoring DOM. Likewise, **persistence after Save + reload**, **undo**,
post-Add **focus** behaviour, and **accessibility-contrast** audits require flows
or tooling outside the authoring editor. **TC_22 (multi-word entry)** is deferred
because the Hidden Word input blocks spaces and accepts letters only, so comma/space
separated entry is not supported by the source input filter; it is documented rather
than asserted. All deferrals are `it.skip()` with explicit reasons in the spec and
in `wordsearch-mapping.json`.
